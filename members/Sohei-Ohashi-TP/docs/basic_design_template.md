## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                                  【Arduino UNO R3】                    【出力】
ジョイスティック（桁選択・値変更） ─────→┐                                     ┌──→ 4Digit-7 Segment（MM:SS）
ボタン1（開始/停止/リセット） ─────────→┤  状態管理・時間計算・入力判定処理   ├──→ active buzzer
ボタン2（通知切替） ─────────────────→┘                                     └──→ blue LED
```

補足説明（図で表現しきれない場合）：
- ジョイスティックで分・秒の各桁を選択し、値を増減する。
- ボタン1で開始→停止→リセットを行う。
- 時間0到達時は、通知モードに応じてブザーまたはLEDで知らせる。
- ボタン2で通知モード（ブザー/LED）を切り替える。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[設定中] ──（ボタン1押下）──→ [計測中] ──（ボタン1押下）──→ [停止中]
   ↑             │                    │                        │
   │             └（時間=0）──→ [通知中] ──（ボタン1押下）──────┘
   └────────────────────────（停止中でボタン1押下:設定値へリセット）

※ ボタン2押下で [設定中/計測中/停止中] から通知モードを切替可能
```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 設定中 | ジョイスティックで桁選択・値増減を受け付け、設定時間を編集 | 7セグに設定値を表示、通知出力OFF |
| 計測中 | 1秒ごとに残り時間を減算（MM:SS） | 7セグに残り時間表示、通知出力OFF |
| 停止中 | カウントダウンを一時停止、次のボタン1押下で設定値に戻す | 停止時点の値を表示、通知出力OFF |
| 通知中 | 時間0到達を通知し、ユーザー操作待ち | ブザーまたはLEDをON（モードに応じる） |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| カウントダウン減算 | 1000ms | **millis** | 入力監視と並行して1秒更新するため |
| 7セグ桁更新（多重化） | 2〜5ms/桁 | **millis** | ちらつきを抑えつつ他処理を止めないため |
| ボタン入力監視 | 常時（10ms程度） | **millis** | 0.1秒以内応答要件を満たすため |
| ジョイスティック反映 | 120ms | **millis** | 誤操作を抑えつつ操作感を保つため |
| 通知点滅（LED時） | 250ms | **millis** | 終了通知を視認しやすくするため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| チャタリング | ボタンを押したとき | 前回入力から50ms未満の入力を検出 | デバウンス処理でスキップ |
| 時間の範囲外 | 設定値が0未満または59:59超相当 | 分秒の演算結果を都度チェック | 9999を表示して設定中へ戻す |
| アナログノイズ | アナログ入力を読んだとき | 閾値付近で値が揺れる | 閾値にヒステリシスを持たせる |
| 残り時間負値化 | 計測更新境界で負値になる | 減算前後の0判定 | 0固定に補正し通知中へ遷移 |
| 通知出力異常 | ブザー/LEDが反応しない | 通知モード時のピン出力確認 | 配線・ピン定義を再確認し安全側停止 |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- ピン競合は見当たらない。
- 状態遷移は設定中/計測中/停止中/通知中で整理されており、復帰ルートも定義されている。
- delay()を使わずmillis()で統一すれば応答性要件に適合しやすい。

**対応した内容：**
- すべての周期処理をmillis()基準で定義。
- 状態遷移図に通知中からの復帰経路を明記。

---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
- ボタンの50msデバウンスは妥当。
- delay()は使わずmillis()運用で問題ない。
- ジョイスティック入力はヒステリシスと入力間引きでノイズ対策が必要。

**対応した内容：**
- A0/A1入力に中立帯と120ms間引きを設計へ追加。

---

### Q3: 要件への対応漏れ確認（任意）

> **【任意】** トレーサビリティ表（Section 5）を書いた場合に実施してください。
>
> 「要件定義書の必須機能と、この基本設計書のSection 5 を比較して、設計漏れや結合テスト計画の抜けを教えてください。」

**AIの回答（要約）：**
- 必須3件はSection 5に対応付け済み。
- 追加機能（通知方式切替）はSection 2とSection 6に補助テストとして反映済み。
